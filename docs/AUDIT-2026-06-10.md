# COMPA 구현 전방위 감사 보고서 (2026-06-10)

목표 기준: `CLAUDE.md` — "제품명 입력 → 4개국(韓·美·日·中) 3년치 할인 이력 → 살지/기다릴지 추천"

## 1. 총평

**전체 진행률: 약 40~45%.** M0(환경)·M1 한국 수집기·M2 프론트엔드의 골격은 실제로 동작하는 수준까지 구현됐다.
그러나 서비스의 핵심 가치 사슬 두 곳이 끊어져 있다:

1. **분류 파이프라인 미연결** — 수집된 `sale_events.event_type`이 항상 NULL로 저장되어,
   추천 로직(`_build_recommendation`)의 "돌발 할인 중 → 지금 사라 / 정기행사 D-n → 기다려라" 분기가
   실데이터로는 절대 발동하지 않는다. 항상 `good_deal` 폴백만 반환된다.
2. **이력 축적 메커니즘 부재** — 모든 스크래퍼가 "현재가 스냅샷"만 수집하고 Celery 정기 수집이 없어서
   ("3년치 할인 이력"은 시간이 지나도 쌓이지 않음), 핵심 데이터 자산이 형성되지 않는다.

## 5. 기능 버그 및 품질 이슈

### 심각 (서비스 가치 직결)
- **event_type 미저장**: collector가 classifier를 호출하지 않아 추천 분기 사장(死藏). → 수집 시 `classify_rule_based` + Claude 폴백 연결 필요.
- **검색 지연**: `/api/products/search`가 `BackgroundTasks` 파라미터를 선언만 하고 **미사용**, `collect_on_demand`를 요청 안에서 동기 대기. 6개 스크래퍼 순차 실행 + Playwright 8초 고정 대기 → 첫 검색이 30~90초 소요 가능.
- **이력 미축적**: 정기 수집(Celery beat) 없음 → 위 1번 항목과 함께 핵심 가치 차단.
- **제품 매칭 부재**: `_get_or_create_product`가 `name_kr` 단일 키 매칭이고 `name_en`에 한국어 이름을 그대로 복사. 국가별 제품명이 다른 4개국 비교가 구조적으로 불가능. `ai/matcher.py` 필요.

### 중간
- 중복 데이터: 24h 캐시 TTL 경과 후 재수집 시 동일 가격이 새 행으로 계속 insert — dedupe 없음.
- 환율 미지원: comparison API가 같은 통화끼리만 정렬/비교 — 4개국 가격 비교라는 목표와 불일치.
- 차단 위험: 쿠팡(단순 httpx GET)·Amazon(HTML 파싱)은 실전에서 봇 차단 가능성 높음. config에 Amazon PA-API 키가 있으나 실제로는 미사용. `core/proxy.py` 미구현.
- 스크래퍼 rate limit이 인스턴스 단위 — 요청마다 새 인스턴스를 만들어 도메인당 1 req/sec 보장이 사실상 무효.
- 추천 로직의 "역대 최고 할인율 95%" 비교가 통화·플랫폼 혼합 데이터 위에서 계산됨 — 의미 왜곡 여지.

### 경미
- Playwright `executable_path="/usr/bin/google-chrome-stable"` 하드코딩 (올리브영·Sephora).
- CORS origin `localhost:5173` 하드코딩.
- `BaseScraper._wait_rate_limit` 내 `import time` 중복.
- 스택 표류: CLAUDE.md는 React 18·Tailwind 3 계열 명시, 실제는 React 19·Tailwind 4·Vite 8 (동작에는 문제 없으나 문서 갱신 필요).
